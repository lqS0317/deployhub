# Feature Specification: DeployHub — K8s 原生运维发布平台

**Feature Branch**: `001-deployhub-platform`  
**Created**: 2026-04-03  
**Status**: Draft  
**Input**: DeployHub 是一个面向小团队（5-20人）的 Kubernetes 原生运维发布平台，管理 <50 个微服务的完整生命周期。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 服务管理与构建 (Priority: P1)

运维工程师（Admin 或 Developer）登录系统后，在服务管理页面创建一个新服务：填写服务名称、关联 Git 仓库和分支、选择目标集群和命名空间、配置容器资源限制和健康检查路径。创建完成后，进入构建中心选择该服务，指定 Git 分支或 Tag，点击"开始构建"。系统在目标集群中创建构建任务，页面实时滚动显示构建日志。构建成功后，系统记录产出的镜像 Tag，该镜像可用于后续发布。

**Why this priority**: 服务注册和镜像构建是整个平台的基础。没有服务定义就无法发布，没有构建产物就无法部署。这是最小可用产品的核心。

**Independent Test**: 可通过创建一个服务并成功构建出镜像来验证。交付价值：团队从此拥有自主可控的镜像构建能力，不再依赖外部 CI。

**Acceptance Scenarios**:

1. **Given** 系统已配置好一个 Git 仓库和一个 K8s 集群, **When** 用户创建服务并填写完整信息, **Then** 服务出现在服务列表中，显示关联的集群和仓库信息
2. **Given** 一个已创建的服务, **When** 用户选择分支并点击构建, **Then** 系统创建构建任务，页面实时显示构建日志流
3. **Given** 构建任务正在执行, **When** 构建成功完成, **Then** 系统记录镜像 Tag 并与 Git commit SHA 关联，构建记录显示为"成功"
4. **Given** 构建任务正在执行, **When** 构建失败, **Then** 日志中显示错误原因，构建记录标记为"失败"，构建资源自动清理
5. **Given** 用户角色为 viewer, **When** 尝试触发构建, **Then** 系统拒绝操作并提示权限不足

---

### User Story 2 — 发布与审批 (Priority: P1)

Developer 在发布管理页面选择一个服务和一个已成功构建的镜像，点击"发起发布"。系统创建发布记录（状态为"待审批"），并向该服务的 Owner 发送审批通知（站内 + 外部渠道）。Owner 在审批中心查看发布详情（包含镜像版本、目标集群、变更对比），选择"通过"或"拒绝"并填写意见。审批通过后，系统自动执行滚动更新，页面实时展示 Pod 逐个替换的进度（如 2/3 Ready）。发布完成后，通知发起人和相关成员。

**Why this priority**: 发布和审批是平台解决的核心痛点——替代无序的 kubectl 操作、引入审批门禁。与 P1-US1 构建配合即形成完整的 CI/CD 闭环。

**Independent Test**: 可通过一次完整的"发起发布 → 审批 → 滚动更新 → 完成通知"流程来验证。交付价值：生产发布从此有审批记录和可视化进度。

**Acceptance Scenarios**:

1. **Given** 一个服务有成功的构建产物, **When** Developer 发起发布, **Then** 系统创建待审批的发布记录并通知 Owner
2. **Given** 一个待审批的发布, **When** Owner 审批通过, **Then** 系统自动开始滚动更新，页面显示实时进度
3. **Given** 一个待审批的发布, **When** Owner 拒绝并填写原因, **Then** 发布记录标记为"已拒绝"，发起人收到通知（含拒绝原因）
4. **Given** 用户是该服务的 Owner, **When** Owner 自己发起发布, **Then** 跳过审批流程，直接执行滚动更新
5. **Given** 用户是 Admin, **When** Admin 发起紧急发布并填写理由, **Then** 跳过审批流程直接执行，系统记录紧急发布的审计日志（含理由）
6. **Given** 滚动更新正在进行, **When** 所有新 Pod 就绪, **Then** 发布状态变为"成功"，通知相关成员

---

### User Story 3 — 回滚 (Priority: P2)

运维人员发现某次发布存在问题，进入该服务的发布历史页面，查看历次发布的镜像版本和时间。选择要回滚到的目标版本，点击"回滚"。系统创建一次新的发布（镜像为目标历史版本），走审批流程（可根据配置跳过紧急回滚的审批）。执行完成后，系统标记本次操作为回滚，记录回滚来源。

**Why this priority**: 回滚是发布的安全网。虽然核心发布流程（P1-US2）已提供价值，但没有回滚能力的发布系统是不完整的。回滚复用发布流程，实现成本低。

**Independent Test**: 可通过"发布 v2 → 发现问题 → 回滚到 v1 → 验证 v1 运行正常"来测试。交付价值：出问题时可在一分钟内恢复到上一个稳定版本。

**Acceptance Scenarios**:

1. **Given** 一个服务有多次发布历史, **When** 用户选择某个历史版本并点击回滚, **Then** 系统创建一次新的发布记录，镜像为历史版本，标记为回滚操作
2. **Given** 回滚发布已创建, **When** 审批通过（或配置为跳过审批）, **Then** 执行滚动更新将服务恢复到目标版本
3. **Given** 回滚完成, **When** 查看发布记录, **Then** 可以看到回滚标记以及回滚来源（从哪个版本回滚）

---

### User Story 4 — 配置管理 (Priority: P2)

运维人员在配置中心为某个服务创建配置模板（使用变量占位符如 `{{ .DatabaseHost }}`），然后分别为 dev/staging/prod 环境填写变量值。编辑完成后，预览渲染结果并与上一版本对比差异。确认无误后下发到目标集群，系统将渲染后的内容写入 K8s ConfigMap 或 Secret。每次变更自动产生新版本，支持查看完整变更历史。

**Why this priority**: 配置管理解决"不同环境配置靠手动维护"的痛点。虽然不阻塞核心发布流程，但对日常运维效率影响显著。

**Independent Test**: 可通过"创建模板 → 填写多环境变量 → 预览渲染 → 下发到集群 → 在集群中验证 ConfigMap 内容"来测试。交付价值：配置变更有版本记录和审计追踪，消除手工配置漂移。

**Acceptance Scenarios**:

1. **Given** 用户进入配置中心, **When** 创建一个配置模板并填入变量占位符, **Then** 模板保存成功并可在列表中查看
2. **Given** 一个已有的配置模板, **When** 为 dev 和 prod 分别填写不同的变量值, **Then** 系统能分别渲染出正确的最终配置内容
3. **Given** 渲染后的配置内容, **When** 用户点击"下发"并选择目标集群和命名空间, **Then** 系统将内容写入 K8s ConfigMap/Secret，并记录下发版本
4. **Given** 一个配置已有多个版本, **When** 用户查看版本历史, **Then** 可以对比任意两个版本的差异（diff 视图）
5. **Given** 模板中包含非法变量引用, **When** 用户预览渲染结果, **Then** 系统提示渲染错误并显示具体的问题位置

---

### User Story 5 — 系统设置与基础设施管理 (Priority: P2)

Admin 在系统设置中管理平台的基础资源：添加 K8s 集群（上传 kubeconfig）、配置 Git 仓库连接（token 或 SSH key）、注册镜像仓库（多云厂商支持）、管理用户和服务成员角色、配置通知渠道（飞书/钉钉/Slack webhook）。所有敏感信息加密存储，集群连接状态可在页面实时检测。

**Why this priority**: 基础设施管理是平台运行的前提条件。虽然优先级标为 P2（因为初始可通过种子数据或 API 调用配置），但在用户自助管理层面是必要的。

**Independent Test**: 可通过"添加集群 → 测试连接成功 → 添加 Git 仓库 → 验证凭证有效 → 注册 Registry"来测试。交付价值：Admin 可自助管理所有基础设施连接，无需直接操作数据库。

**Acceptance Scenarios**:

1. **Given** Admin 进入集群管理, **When** 上传 kubeconfig 并保存, **Then** 凭证加密存储，页面显示集群信息（名称、环境、API Server 地址）
2. **Given** 已添加一个集群, **When** 点击"测试连接", **Then** 系统尝试连接集群并返回成功/失败结果
3. **Given** Admin 进入 Git 仓库管理, **When** 添加仓库并提供 token, **Then** token 加密存储，系统验证仓库可达
4. **Given** Admin 进入用户管理, **When** 为某用户分配某服务的 developer 角色, **Then** 该用户可在该服务上触发构建和发布
5. **Given** 非 Admin 用户, **When** 尝试访问系统设置, **Then** 系统拒绝并提示权限不足

---

### User Story 6 — 服务导入 (Priority: P3)

团队已有运行在 K8s 上的服务，希望批量导入到 DeployHub 中管理。用户在服务管理页面选择"导入服务"，上传 K8s Deployment + Service 的 YAML 文件（或使用简化的自定义格式）。系统解析文件内容，提取出服务名、镜像、副本数、端口、资源限制等信息，展示预览列表。用户补充缺失信息（如关联的 Git 仓库、负责人），确认后批量创建服务记录。

**Why this priority**: 导入功能提升存量服务的接入效率，但不阻塞核心流程（新建服务已可用）。

**Independent Test**: 可通过"上传一组 YAML 文件 → 预览解析结果 → 确认导入 → 验证服务出现在列表中"来测试。交付价值：存量几十个服务无需逐一手动创建。

**Acceptance Scenarios**:

1. **Given** 用户有一组 K8s Deployment YAML, **When** 上传文件, **Then** 系统解析出服务名、镜像、副本数等信息并展示预览
2. **Given** 解析后的预览列表, **When** 用户补充 Git 仓库和负责人信息并确认, **Then** 系统批量创建服务记录
3. **Given** 上传的 YAML 中有重复的服务名, **When** 系统检测到重复, **Then** 提示用户选择"跳过"或"覆盖"
4. **Given** 上传文件格式不合法, **When** 系统解析失败, **Then** 提示具体的解析错误位置和原因

---

### User Story 7 — 通知与审计 (Priority: P3)

用户在通知中心查看所有待处理和历史通知（审批请求、构建完成、发布结果等），支持标记已读。系统同时通过配置的外部渠道（飞书/钉钉/Slack）推送关键事件。Admin 可在审计日志中搜索和筛选所有操作记录，追踪"谁在什么时间做了什么操作"。

**Why this priority**: 通知和审计是锦上添花的功能，核心流程中已内嵌基础通知能力（如审批通知）。独立的通知中心和审计查询界面提升运维体验但不阻塞核心功能。

**Independent Test**: 可通过"触发一次构建 → 验证站内通知出现 → 验证外部渠道收到消息 → 在审计日志中搜索该操作"来测试。交付价值：操作可追踪，关键事件不遗漏。

**Acceptance Scenarios**:

1. **Given** 一次构建完成, **When** 用户打开通知中心, **Then** 看到构建完成的通知记录
2. **Given** 配置了飞书通知渠道, **When** 有待审批的发布, **Then** Owner 同时收到站内通知和飞书推送
3. **Given** Admin 进入审计日志页面, **When** 按时间范围和操作类型筛选, **Then** 显示匹配的操作记录列表（操作人、时间、操作类型、目标资源、详情）
4. **Given** 用户有多条未读通知, **When** 点击"全部标记已读", **Then** 所有通知标记为已读，未读计数归零

---

### Edge Cases

- **构建超时**：构建任务运行超过配置的最大时长时，系统自动终止任务、标记为失败、清理资源并通知用户
- **发布期间集群不可达**：执行滚动更新时目标集群断连，系统标记发布为"失败"并保留当时的 Pod 状态快照信息
- **并发发布冲突**：同一服务同一集群有正在进行的发布时，拒绝新的发布请求并提示"该服务当前有正在执行的发布"
- **审批超时**：待审批的发布超过配置的审批时限后，自动标记为"已过期"并通知发起人
- **配置模板渲染失败**：变量值缺失或格式不匹配时，渲染阶段拦截并给出具体错误提示，禁止下发无效配置
- **kubeconfig 过期**：操作集群前检测凭证有效性，失效时标记集群为 inactive 并提醒 Admin 更新
- **构建资源清理**：构建完成或失败后，自动清理集群中的构建任务资源，避免资源泄漏
- **WebSocket 断线**：前端断线后自动重连，重连后请求补发缺失的日志/进度事件
- **批量导入部分失败**：导入过程中部分服务解析失败时，成功的继续创建，失败的汇总报告

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持用户通过本地账号（用户名/密码）和 OAuth2（多 Provider）登录
- **FR-002**: 系统 MUST 实现基于服务维度的角色控制（owner / developer / viewer），每个角色有明确的操作权限边界
- **FR-003**: 系统 MUST 支持管理多个 K8s 集群，每个集群的凭证独立加密存储，支持连接状态检测
- **FR-004**: 系统 MUST 支持管理多个 Git 仓库连接（GitHub / GitLab / 其他），凭证加密存储
- **FR-005**: 系统 MUST 支持管理多个容器镜像仓库（ECR / ACR / TCR / Harbor 等），凭证加密存储
- **FR-006**: 系统 MUST 支持创建服务并关联 Git 仓库、目标集群、镜像仓库，配置部署参数（副本数、资源限制、端口、健康检查等）
- **FR-007**: 系统 MUST 支持通过上传 K8s YAML 或简化格式文件批量导入服务，遵循"解析 → 预览 → 确认"流程
- **FR-008**: 系统 MUST 支持在 K8s 集群内触发镜像构建任务（无 Docker daemon 依赖），构建日志实时推送到前端
- **FR-009**: 构建产出的镜像 Tag MUST 与 Git commit SHA 关联，确保版本可溯源
- **FR-010**: 系统 MUST 支持发起发布请求，Developer 发起时需 Owner 审批，Owner 自己发起时跳过审批，Admin 可紧急跳过（需理由）
- **FR-011**: 发布 MUST 使用滚动更新策略，发布进度（Pod 就绪数/总数）实时推送到前端
- **FR-012**: 系统 MUST 支持回滚到任意历史版本，回滚复用发布流程（含审批），标记为回滚操作并记录来源
- **FR-013**: 系统 MUST 支持配置模板管理：创建模板（变量占位符）、按环境填写变量值、预览渲染结果、版本对比、下发到集群
- **FR-014**: 配置下发 MUST 将渲染后的内容写入目标集群的 ConfigMap 或 Secret，每次下发产生新版本记录
- **FR-015**: 系统 MUST 对所有写操作产生审计日志，记录操作人、操作类型、目标资源、操作详情（脱敏）、IP 地址和时间
- **FR-016**: 系统 MUST 支持站内通知，并可通过外部渠道（飞书 / 钉钉 / Slack webhook）推送关键事件
- **FR-017**: 所有敏感数据（kubeconfig、Git 凭证、Registry 凭证、Secret 值）MUST 加密后存储，禁止明文落库
- **FR-018**: 系统 MUST 在构建/发布完成或失败后自动清理集群中的临时资源（构建任务、临时 Pod）
- **FR-019**: 系统 MUST 在同一服务同一集群已有正在执行的发布时，拒绝新的发布请求
- **FR-020**: 系统 MUST 支持构建超时检测和自动终止，超时阈值可配置

### Key Entities

- **User**: 系统用户，支持本地和 OAuth 登录，全局角色为 admin 或 member
- **ServiceMember**: 用户与服务的关系，角色为 owner / developer / viewer，决定对该服务的操作权限
- **Cluster**: K8s 集群，包含连接信息和加密凭证，区分环境（dev / staging / prod）
- **GitRepo**: Git 仓库连接信息和加密凭证，支持多平台
- **Registry**: 容器镜像仓库连接信息和加密凭证，支持多云厂商
- **Service**: 核心实体，关联 Git 仓库、目标集群、镜像仓库，包含完整的部署配置参数
- **Build**: 镜像构建记录，追踪构建任务状态和日志，产出镜像 Tag
- **Deployment**: 发布记录，包含目标镜像版本和前一版本（用于回滚），追踪发布状态和进度
- **Approval**: 审批记录，关联发布请求，记录审批人意见和决定
- **ConfigTemplate**: 配置模板，使用变量占位符定义配置结构
- **ConfigEnvValue**: 每个环境的变量具体值，关联模板和集群
- **ConfigVersion**: 渲染后的配置内容快照，支持版本对比
- **ConfigDeployment**: 配置下发记录，追踪哪个版本被下发到哪个集群
- **Notification**: 站内通知记录，关联触发事件和目标用户
- **AuditLog**: 操作审计记录，仅追加不删除

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在 5 分钟内完成"创建服务 → 触发构建 → 发起发布 → 审批通过 → 发布完成"的全流程
- **SC-002**: 构建日志从产生到前端显示的延迟不超过 2 秒（用户感知为实时）
- **SC-003**: 发布进度更新（Pod 就绪状态变化）从集群到前端显示的延迟不超过 3 秒
- **SC-004**: 回滚操作从发起到执行完成不超过 3 分钟（含审批环节）
- **SC-005**: 系统可同时管理至少 50 个服务和 5 个 K8s 集群而不出现性能退化
- **SC-006**: 所有写操作 100% 产生审计记录，审计日志可通过时间/操作类型/操作人检索
- **SC-007**: 配置变更可在 30 秒内完成"修改变量 → 预览渲染 → 下发到集群"
- **SC-008**: 新团队成员在无人指导下 15 分钟内可完成首次服务创建和构建（可用性指标）
- **SC-009**: 90% 的审批请求在发出通知后 10 分钟内被处理（依赖外部通知渠道可达性）
- **SC-010**: 存量服务通过批量导入功能，30 个服务可在 10 分钟内完成导入

## Assumptions

- 目标用户为具备 K8s 基础知识的运维和开发人员，熟悉 Deployment、ConfigMap、Namespace 等概念
- 用户通过现代浏览器（Chrome/Firefox/Edge 最近两个大版本）访问系统，不考虑移动端适配
- 部署环境已有可用的 K8s 集群，DeployHub 自身也运行在 K8s 上
- 外部依赖（Git 平台、镜像仓库、通知渠道）已存在且可达，系统仅做对接而非搭建
