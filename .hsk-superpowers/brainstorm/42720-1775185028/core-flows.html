核心业务流程

构建、发布、回滚的完整流转

🔨 构建流程
用户选择服务
指定分支/Tag
→
后端校验权限
developer+
→
创建 Build 记录
status: pending
→
创建 Kaniko Job
K8s Pod 构建镜像
→
Watch Pod 状态
WebSocket 推送日志
→
构建完成
记录 image_tag
Kaniko Job 细节：创建一个 K8s Job，挂载 Git 凭证 Secret → Kaniko init container 拉取源码 → Kaniko 主容器根据 Dockerfile 构建 → 推送到配置的 Registry → 后端通过 Watch Job 状态感知完成/失败。Job 完成后自动清理。
🚀 发布流程（含审批）
选择服务+镜像
选构建产物
→
创建 Deployment
status: pending_approval
→
通知服务 Owner
站内 + IM 推送
→
Owner 审批
approve / reject
→
执行滚动更新
client-go patch
→
Watch Rollout
实时进度推送
→
发布完成
通知相关人
审批规则：发起人是服务 owner 时，跳过审批直接发布。发起人是 developer 时，需要至少一个 owner 审批通过。Admin 角色可以跳过审批（紧急发布）。
进度追踪：后端通过 client-go Watch Deployment 的 status.conditions 和 readyReplicas，计算进度百分比，通过 WebSocket 实时推送到前端。
⏪ 回滚流程
查看发布历史
选择回滚目标版本
→
创建新 Deployment
image = 目标版本
→
审批（可配置）
紧急回滚可跳过
→
执行滚动更新
替换为旧镜像
→
回滚完成
标记 rolled_back
设计要点：回滚本质是一次"使用旧版本镜像的新发布"，复用发布流程。在 Deployment 记录中标记 is_rollback=true 和 rollback_from_id 指向被回滚的发布记录，便于审计追踪。
📥 服务导入流程
方式一：K8s YAML 导入
1. 上传 Deployment + Service YAML 文件
2. 解析出 name, image, replicas, ports, resources, env 等
3. 映射到 Service 实体字段
4. 用户预览并补充缺失信息（Git 仓库、负责人等）
5. 确认后批量创建 Service 记录
方式二：简化配置导入
1. 上传自定义 YAML/JSON 配置文件
2. 格式精简：name, git_url, branch, image, port 等
3. 支持批量定义多个服务
4. 自动关联已配置的 Git 仓库和集群
5. 用户预览确认后批量创建
两种方式共同点：都经过"解析 → 预览 → 确认"三步，不会盲目导入。导入时自动检测重复服务名，支持跳过或覆盖。
